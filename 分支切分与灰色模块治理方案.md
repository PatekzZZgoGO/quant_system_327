# 分支切分与灰色模块治理方案

## 问题背景

当前真正的难点，不在于“现在能不能分支”，而在于“分支切出去之后怎么持续演进”。

核心矛盾主要有三点：

- 灰色模块如果直接放进某个分支维护，后面另一边也会继续使用，最终就会逐渐漂移。
- 灰色模块如果直接放入公共部分，公共层又会被具体业务语义污染。
- 一旦两边都继续演进，共有部分的更新又不能自然同步，维护成本会持续升高。

因此，这个问题不能只靠“分支怎么切”解决，更关键的是“切完之后怎么治理”。

## 基本判断

不应该把这件事理解成“两个长期并行开发分支共享一大块代码”。

更稳的方式是：

- 不要用 Git 分支承载长期共享代码演进，分支只承载阶段性分流。
- 长期应该有一个稳定主干承载公共基础。
- 分析决策方向与交易方向分别在公共基础之上演进。
- 灰色模块通过逐步拆分处理，而不是长期靠人工同步。

也就是说，灰色部分不能一直灰着，必须给出明确的后续治理策略。

## 三类模块治理模型

### 1. 真正公共的模块

这类模块应留在公共主干中。

特点：

- 两边都会长期依赖。
- 业务语义较低。
- 抽象相对稳定。

当前典型例子：

- `data` 中的共享 facade
- `features` 中的通用计算能力
- `utils` 中的元数据、运行记录等基础能力

治理原则：

- 只能放抽象稳定的接口和通用能力。
- 不允许放“为了某一边方便”而引入的场景逻辑。

### 2. 明显私有的模块

这类模块应直接归属某个方向，不再犹豫是否进入公共层。

当前典型例子：

- `execution`
- `risk`
- `portfolio`
- `live`

治理原则：

- 谁的业务语义更强，谁负责维护。
- 另一边如果需要，只能通过结果消费或重新实现，不直接反向耦合。

### 3. 灰色模块

这类模块最关键，不应该长期共享整块实现，而应该只共享边界。

当前典型例子：

- `backtest`
- `models/alpha`
- `strategies`

推荐处理方式不是“整目录放哪边”，而是拆成两层：

- 公共层：协议、输入输出格式、最小可复用组件
- 私有层：各自分支里的具体编排和业务解释

换句话说，灰色模块不要共享“整块实现”，只共享：

- 数据结构
- 接口约定
- 最小算法骨架
- 测试基线

不要共享：

- 场景参数推导
- 业务编排
- 交易语义
- 产品化输出逻辑

## 公共层准入规则

后续只有同时满足以下条件的能力，才允许进入公共层：

- 两边都会长期使用
- 短期不会被单边需求频繁改形
- 不带明显分析产品或交易运行时语义

如果不能同时满足这三条，就不应进入公共层。

## 灰色模块的落地拆分方式

对于灰色模块，不建议做“目录整体归属”，而建议做“接口切分”。

以 `backtest` 为例：

- 公共部分：结果结构、统计口径、纯分析组件
- 交易侧：执行延迟、成交模型、持仓与订单语义
- 分析侧：研究回测编排、策略评估输出

这个原则同样适用于 `models/alpha` 和 `strategies`：

- 公共部分保留最小协议、最小输出约定与基础测试
- 分支私有部分承接各自方向的业务解释和具体实现

## 后续新增需求的判断原则

以后新增任何能力时，都先问一句：

“这是公共能力，还是某个方向的业务便利？”

如果答案是后者，就禁止塞回公共层。

## 共享更新的处理原则

公共层更新不能依赖两个分支之间互相复制、手工同步或长期 cherry-pick。

更合理的结构应是：

- 一个 `shared` 主干
- `analysis` 在其上演进
- `trading` 在其上演进

进一步，如果后续共享部分继续稳定，还可以考虑将 `shared` 独立成单独包或子仓库。

否则，“共有部分更新又不能同步”的问题一定会不断重复出现。

## 直接结论

灰色部分不能靠“放哪个分支”来解决。

真正可行的方式只能是：

- 拆接口
- 拆职责
- 只共享边界
- 不共享整块业务实现

## 建议执行顺序

最现实的执行路线如下：

1. 先切分支。
2. 但不要马上迁移大目录。
3. 先选择一个灰色模块试刀，优先 `backtest`。
4. 将其拆成三部分：
   - `shared contract / minimal core`
   - `analysis-owned orchestration`
   - `trading-owned runtime semantics`

这样可以先验证这套拆分方法是否真正可行，再决定是否推广到 `models/alpha` 与 `strategies`。
